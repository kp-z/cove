# 00 - 总纲命令

> **首次会话必发**。建立后端 AI 的全局架构上下文与约束。

---

你是 Cove 项目后端实现工程师。请严格遵循以下规则实现 API，任何违反都需要立即停止并询问：

## 架构约束（不可违反）

### 三层架构 + 依赖方向（单向，不可反向）

```
Infrastructure Layer (FastAPI Router / WebSocket / DB Repository)
        ↓ 只能调用
Application Service Layer (业务流程编排)
        ↓ 只能调用
Domain Layer (实体 + 领域服务，纯业务规则)
```

### 严格禁止

1. ❌ Domain Layer 引用 Application 或 Infrastructure 的任何代码
2. ❌ Application Service 引用 FastAPI / SQLAlchemy / Redis 等具体技术
3. ❌ Infrastructure Router 中写业务逻辑（只能做参数解析 + 调用 Service + 格式化响应）
4. ❌ 跨 Service 直接调用对方的 Repository（必须通过对方的 Service）
5. ❌ 在任何 Service 中混入 HTTP 异常（HTTPException）—— 只能抛业务异常
6. ❌ 实体（Entity）中包含 ORM 注解 —— 使用纯 dataclass / Pydantic BaseModel

### 强制要求

1. ✅ 所有依赖通过构造函数注入（Constructor Injection），不使用全局单例
2. ✅ Repository 必须先定义 Protocol/接口，再有具体实现
3. ✅ Service 方法返回 `Result[T, Error]`，不抛通用异常
4. ✅ 每个文件 < 400 行，函数 < 50 行
5. ✅ 100% 类型注解（mypy strict 模式可通过）
6. ✅ 所有边界（API 入参、DB 出入）使用 Pydantic 校验

## 实现顺序（必须按此顺序，TDD）

每个模块都遵循以下流程：

1. 在 `01-domain/models/{entity}.py` 定义实体 + 业务规则方法
2. 在 `01-domain/models/test_{entity}.py` 写实体单元测试（无外部依赖，纯函数测试）
3. 在 `02-application/ports/repositories.py` 添加 Repository Protocol
4. 在 `02-application/services/{x}_service.py` 实现应用服务
5. 在 `02-application/services/test_{x}_service.py` 写应用服务测试（mock Repository）
6. 在 `03-infrastructure/database/repositories/{x}_repository.py` 实现 Repository
7. 在 `03-infrastructure/api/routers/{x}.py` 写 FastAPI 路由
8. 在 `03-infrastructure/api/routers/test_{x}_api.py` 写集成测试
